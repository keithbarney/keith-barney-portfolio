Case Study 01

Creating Experian's First Design System

Big picture · High-level expertise · Wide range

Case Study 01

Problem

No system.
No standards.
No alignment.

Multiple domain teams independently creating UI patterns. A style guide offered guidance but no enforceable standards — leading to inconsistency, ongoing debate, and slow decision-making.

Inconsistency No governance Fragmented patterns Slow shipping

Case Study 01

What I Built

Directly Owned

  • Buy-in communication with Design & Engineering Managers
  • First reusable component libraries — iOS, Android, Web
  • Slack support channels + intake workflows
  • Base/alias token system architecture
  • Typography scale
  • Design QA process for system alignment

Influenced

  • Designers creating domain-specific reusable components
  • Organization-wide shift toward systems thinking

Case Study 01

Hard Calls

Sold the vision to skeptics

Engineering didn't believe a design system was needed. I sold the value: centralized decisions, reusable code, faster shipping, baked-in accessibility.

Killed my own initiative

Design QA couldn't scale without more resources. I pushed ownership back to domain leads — a hard lesson in knowing when to let go.

Educated through friction

Taught designers that overriding and detaching component instances created downstream confusion for developers.

Case Study 01

Impact

0 → 1 Design system created from scratch — first at Experian
3 platforms iOS, Android, Web — unified component libraries
Company-wide mandate Head of design mandated all new projects use Atlas
Team formalized Keith + 2 native designers → first official DS team
Less custom code, fewer bugs Engineering impact from centralized, reusable components

Case Study 02

Atlas Component Merge Across Platforms

On the ground · Change management at scale

Case Study 02

Problem

Same components.
Different everywhere.

Platform-specific libraries evolved independently — Web, iOS, Android. The web library had separate mobile, tablet, and desktop breakpoints. Fragmentation led to duplicated patterns, inconsistent behavior, and mounting maintenance overhead.

Platform fragmentation Duplicated patterns Maintenance burden

Case Study 02

Approach

Co-owned with Caleb. We divided the component list, created branches, merged and released in waves over 2 months.

Formal deprecation process

Created a 4-step deprecation workflow: announce → migrate → sunset → remove.

Communication at every level

Slack announcements, Loom walkthroughs for breaking changes, 1:1 meetings with designers for check-ins and demos.

Stakeholder buy-in chain

Sold Brett first, then Adam. Gave Adam a process and timeline for tracking progress to leadership.

Case Study 02

Impact

1 component, all platforms Same component works across iOS, Android, Web — fewer Figma frames
Centralized decisions One source of truth replaced platform-specific interpretations
Team skill uplift Designers' Figma skills greatly increased through migration process
Reduced switching cost No more context-switching between platform-specific libraries

Case Study 03

Atlas Audit

Solving quality · Measurable outcomes · Autonomous initiative

Case Study 03

Problem

Adoption was invisible.

Four UI systems coexisted in production: Legacy, Next, Atlas, and SDUI. Nobody could see what was actually running. Adoption was abstract, slow, and impossible to operationalize or prioritize.

4 UI systems No visibility Abstract adoption Tech debt hidden

Case Study 03

What I Built

An internal tool that visualizes which UI system every element in production belongs to — turning abstract "adoption" into a concrete, prioritized backlog.

Directly Owned

  • Conceived and owned the idea end-to-end
  • Created Figma prototype for developer handoff
  • Collaborated with engineers for feedback loops
  • Pursued autonomously — Brett and Adam trusted me to run with it

Influenced

  • Inspired team members to create their own internal tools
  • Shifted developer mindset — they started bringing ideas forward
  • Proved I identify and solve problems, not just work tickets

Case Study 03

Impact

Tech debt → visible Made invisible UI system debt concrete and actionable
Saved contract spend Surfaced work for offshore team running low on tasks — prevented waste
Adoption operationalized Abstract "adoption" became prioritized work items with clear ownership
Culture shift Team started proactively building internal tools and bringing ideas forward
